## 铁律

**没有证据就不能声称通过。** 违反字面就是违反精神。

**"没有新鲜验证证据就不许声称完成。"** 如果当前消息中没有执行过验证命令，就不许断言某项通过。

## 门禁函数

做出任何状态声明前：

1. **识别**：哪个命令能证明你的声明
2. **运行**：完整执行该命令
3. **读取**：完整输出，包括退出码和失败数
4. **验证**：输出是否真正确认了你的断言
5. **然后**才做出声明并附带证据

跳过任何一步 = 不诚实。

## Red Flag

以下措辞意味着未验证就声称：

- "应该通过了" / "probably passes"
- "似乎没问题" / "seems fine"
- "我觉得可以" / "I think it works"
- 表达满意在验证之前
- 信任 agent 报告而无独立确认

**发现Red Flag？** 立即停止，运行实际命令，读取输出，然后才声称。

## 执行流程

### Step 1：Spec 覆盖检查

- [ ] 浏览 spec 中的每个部分/需求
- [ ] 可以指向实现它的 task
- [ ] 列出任何缺口

### Step 2：占位符扫描

- [ ] 搜索 "TBD"、"TODO"、"implement later"、"fill in details"
- [ ] 修复所有占位符

### Step 3：类型一致性检查

- [ ] 后续 task 中使用的类型、方法签名和属性名与早期 task 定义匹配
- [ ] 函数命名一致（如 clearLayers vs clearFullLayers）

### Step 4：运行编译和测试

- [ ] BUILD_CMD 通过
- [ ] VET_CMD 通过
- [ ] TEST_CMD 通过

### Step 4.5：测试持久化与回归验证

- [ ] 单元测试文件已持久化到项目标准测试目录（非临时文件）
- [ ] 集成测试文件已持久化到项目标准测试目录（非临时文件）
- [ ] 回归测试已运行，无新增代码引起的失败
- [ ] 预先存在的失败已标记为 WARN 并记录在测试报告中

### Step 5：代码质量检查

- [ ] 遵守编码红线（读取 constitution.md，逐一检查）

- [ ] 日志格式规范

### Step 6：功能验证

- [ ] spec.md 中的所有功能已实现
- [ ] 功能点可正常使用
- [ ] 输出/响应符合预期
- [ ] 错误处理正确

## 约束

- 所有检查必须通过才能提交
- 任何一项失败都需要先修复
- 不允许跳过检查

## progress.md 更新

**开始执行时**：更新 `specs/<date+feature>/progress.md`，将 Step 5 状态设为 `▶ 进行中`，**开始时间填写当前时间（HH:mm 格式，如 14:30）**；在 Skill 调用记录中追加一行，时间列填写当前时间。

**执行完成时**：将 Step 5 状态更新为 `✅ 完成`，**完成时间填写当前时间（HH:mm 格式）**；在 Skill 调用记录中更新对应行结果为 `✅ 已完成`，时间列填写完成时的时间。

**验证失败时**：将 Step 5 状态更新为 `❌ 失败`，完成时间填写失败时的时间，备注列填写失败原因；在 Skill 调用记录中更新对应行结果为 `❌ 失败`。

**关键：时间必须填入实际的 HH:mm 数值（如 14:30），禁止填入字面量 "HH:mm"。**

## 完成条件与下一步

**验证通过后：** 更新 `specs/<date+feature>/progress.md`（按上述规则填写完成时间），触发 index-update（loom-index-update skill）

**验证未通过：** 更新 `specs/<date+feature>/progress.md`（按上述规则标记失败），输出修复指令后回到 executing 阶段。**禁止全量重新执行 Step 4**，只派发 implementer（修复模式）修复 BLOCKER 对应的问题。

#### 验证失败时的修复指令

当验证发现 BLOCKER 时，必须在验证报告中输出结构化的修复指令，供编排器直接传递给 implementer 修复模式：

```markdown
## 修复指令

### 修复项 1
- **问题**：<具体问题描述>
- **文件**：<文件路径>
- **位置**：<函数名/行号/区域>
- **严重度**：阻断
- **修复方向**：<具体的修复方向>

### 修复项 2
- **问题**：<具体问题描述>
- **文件**：<文件路径>
- **位置**：<函数名/行号/区域>
- **严重度**：阻断
- **修复方向**：<具体的修复方向>
```

修复指令只包含阻断问题，不包含警告和建议。编排器将提取修复指令，派发 implementer（修复模式），只传递修复指令 + subagent-context.md，不重新传递完整 task 上下文和 spec 全文。

## 流程图

```
Spec覆盖 → 占位符扫描 → 类型一致性 → 编译验证 → 静态分析 → 测试验证
→ 测试持久化检查 → 回归测试验证 → 代码质量 → 功能验证 → 文档同步 → 输出报告
                                          ├→ 全部通过 → 完成
                                          └→ 有阻断问题 → 输出修复指令
                                              → 派发implementer(修复模式)
                                              → 修复后重新验证（循环）
```
