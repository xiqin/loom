# 设计自检清单

## 使用场景

在 brainstorming 完成后，写入 spec.md 之前，使用此清单检查设计。

## 检查项

### 1. 占位符扫描
- [ ] 有任何 "TBD"、"TODO"、"implement later" 吗？
- [ ] 有任何未完成的部分吗？
- [ ] 有任何模糊的需求吗？

**如果发现：** 修复它们，不要留下占位符。

### 2. 内部一致性
- [ ] 各部分之间没有矛盾吗？
- [ ] 架构描述与功能描述一致吗？
- [ ] 接口定义与数据模型匹配吗？
- [ ] 流程图与文字描述一致吗？

**如果发现：** 内联修复矛盾。

### 3. 范围检查
- [ ] 设计聚焦到单个实现计划吗？
- [ ] 需要分解成多个子项目吗？
- [ ] 有没有超出初始需求的功能？

**如果发现：** 分解或移除超出范围的内容。

### 4. 歧义检查
- [ ] 任何需求可能有两种解释吗？
- [ ] 所有术语都明确定义了吗？
- [ ] 所有缩写都解释了吗？

**如果发现：** 选择一种解释并明确说明。

### 5. 技术可行性
- [ ] 技术方案可行吗？
- [ ] 依赖的资源可用吗？
- [ ] 性能指标可达吗？

**如果发现：** 调整方案或技术选型。

### 6. YAGNI 检查
- [ ] 有不必要的功能吗？
- [ ] 有过度设计的部分吗？
- [ ] 可以简化吗？

**如果发现：** 无情删除，保持简单。

## 总结

- **通过：** 可以写入 spec.md
- **未通过：** 修复问题，重新检查

**不需要重新审查 — 修复后继续。**
