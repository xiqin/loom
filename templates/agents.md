1. `.loom/rules/constitution.md`：项目原则、技术栈、验证命令和红线。

4. `.loom/memory/MEMORY.md`：长期记忆、踩坑记录和用户偏好。

## 开发流水线（强制）

收到功能需求或非平凡开发任务时，**必须严格执行以下协议，禁止跳步、禁止凭记忆假设步骤**：

### 第一步：判断任务类型（必须在动笔前完成）

根据用户描述，从以下类型中选择一个，**必须明确告知用户**：

| 类型 | 适用场景 |
|------|----------|
| `feature` | 新功能开发 |
| `bugfix` | 已定位的 bug 修复 |
| `hotfix` | 生产紧急问题 |
| `refactor` | 代码重构 |
| `chore` | 依赖升级、配置调整、文档更新等低风险改动 |
| `pm-prototype` | PM 需求到原型：需求 → spec → HTML 原型（无编码） |

如无法判断，默认使用 `feature`，并告知用户。

### 第二步：读取并展示当次流水线（必须执行）

**立即读取 `.loom/workflow.yaml`**，找到对应类型的 `pipelines.<type>.steps`，将完整步骤以如下格式展示给用户，然后等待用户确认后再继续：

```
📋 当前流水线：<type>
─────────────────────────────
# | 阶段 | Skill | Gate
1 | <step.description> | <step.skill> | <gate 或 —>
2 | ...
─────────────────────────────
输入 ok 确认，或告诉我需要调整的步骤。
```

**禁止在用户确认前执行任何步骤。**

### 第三步：按步骤逐一执行

确认后，严格按 `.loom/workflow.yaml` 中该流水线的 `steps` 顺序执行，每个步骤必须：

1. **加载对应 skill**（`step.skill` 字段指定的 skill）。
2. **执行该 skill 的完整流程**，产出对应产物。
3. **遇到 `gate: human-approval` 时，必须停下来等待用户明确确认**，禁止自动跳过。
4. **更新进度文件** `specs/<date+feature>/progress.md`：
    - 开始时：`▶ 进行中`，填写实际开始时间 HH:mm
    - 完成时：`✅ 完成`，填写实际结束时间 HH:mm，备注栏注明产物路径
    - 失败时：`❌ 失败`，填写时间，备注栏注明失败原因
5. **完成后明确告知用户**本步骤产物，再进入下一步。

### 特殊规则

- **验证未通过**：回到 `executing` 步骤修复，不重跑整个流水线。
- **小改动**（单文件修复、配置调整）：可跳过流水线，但必须明确告知用户"本次跳过流水线，原因：xxx"。
- **workflow.yaml 不可读**：停止执行，告知用户文件缺失，不得凭记忆假设流水线内容。
- **步骤中途失败**：更新 progress.md 为 `❌ 失败`，向用户报告失败原因和建议，等待指示，不自动重试。

## 工作方式

- 先理解需求和现有约定，再做最小必要改动。
- subagent/并行执行只用于相互独立、边界清楚的任务；主线阻塞工作由当前 agent 负责。
- 外部服务、浏览器、数据库、CI 或 issue 系统优先通过 MCP、插件或本地命令访问。

## 完成前检查

交付前确认：

1. 相关验证命令已经运行，或明确说明无法运行的原因。
2. 新增/删除的路由、模块、命令、关键约定已同步索引（`loom index`：codegraph 可用时委派 codegraph sync，否则更新 `.loom/index/engineering-index.md`）。
3. 重要踩坑、用户偏好或跨会话决策已记录到 `.loom/memory/MEMORY.md`。
